# План сервиса для выбора и трекинга фильмов и сериалов

Дата версии: 6 июня 2026
Статус: проект для личного теста
Формат запуска: бесплатный сайт для небольшой закрытой аудитории
Ожидаемая аудитория: до 50 друзей
Коммерческая цель: отсутствует
Основной хостинг: Vercel
Основная база данных: облачный Postgres на Neon
Основной источник фильмов и сериалов: TMDB API

## 1. Краткое описание идеи

Сервис помогает пользователю выбрать, что посмотреть одному или вместе с друзьями, а затем сохранить историю просмотров, оценки, статусы, прогресс по сериалам и личные коллекции.

Проект не должен конкурировать с крупными каталогами фильмов. Каталог нужен как база, но основная ценность сервиса в другом:

- быстро найти фильм или сериал;
- сохранить в личный список;
- понимать, что уже просмотрено, что запланировано, что брошено;
- отслеживать сериалы по сезону и эпизоду;
- создавать свои коллекции;
- выбирать фильм вместе с друзьями через комнату голосования.

Рабочее позиционирование:

Сервис для выбора фильмов с друзьями и ведения личного списка просмотров.

## 2. Главная гипотеза проекта

Людям удобно использовать простой сайт, где можно:

1. Найти фильм или сериал.
2. Добавить его в свой список.
3. Отметить статус просмотра.
4. Поставить оценку.
5. Для сериала сохранить текущий сезон и эпизод.
6. Создать комнату и выбрать фильм вместе с друзьями.

Если пользователи возвращаются через несколько дней и продолжают добавлять тайтлы, значит идея имеет смысл для дальнейшего развития.

## 3. Ограничения первой версии

Проект делается не для коммерции, а для личного теста на небольшой группе друзей.

Ограничения:

- максимум около 50 пользователей;
- только веб-сайт, без мобильного приложения;
- без оплаты, рекламы, подписок и affiliate-ссылок;
- без публичного продвижения;
- без сложной социальной сети;
- без копирования полной базы фильмов в свою БД;
- без использования IMDb и Кинопоиска как обязательных источников данных;
- основа данных - TMDB API;
- интерфейс сначала можно сделать простым, но понятным.

## 4. Принятые допущения

Пока нет финального названия, поэтому в документе сервис называется условно `MovieRoom`.

Для первой версии принимаются такие решения:

- язык интерфейса: русский;
- типы контента: фильмы и сериалы;
- оценка: от 1 до 10;
- авторизация: через Google OAuth на базе Auth.js;
- хостинг: Vercel Hobby;

- ORM: Prisma;
- стили: Tailwind CSS;
- фронт и бэк: один Next.js проект;
- роутинг: Next.js App Router;
- API внутри проекта: Route Handlers в папке `app/api`;
- деплой: через GitHub и Vercel;
- первая аудитория: только друзья по прямой ссылке.

Что можно изменить позже:

## 5. Выбранный стек

### 5.1. Общая архитектура

Проект делается как fullstack-приложение на Next.js.

Один проект содержит:

- страницы сайта;
- серверные обработчики запросов;
- работу с TMDB API;
- работу с базой данных;
- авторизацию;
- UI-компоненты.

Схема:

```text
Пользователь
  -> сайт на Vercel
    -> Next.js страницы
    -> Next.js Route Handlers
      -> TMDB API
      -> Neon Postgres
```

### 5.2. Технологии

Frontend:

- Next.js;
- React;
- TypeScript;
- Tailwind CSS.

Backend:

- Next.js Route Handlers;
- Server Actions, если будет удобно для форм;
- Auth.js для авторизации;
- Prisma для работы с Postgres.

Database:

- Neon Postgres;
- Prisma migrations;
- локальное хранение только пользовательских данных и минимального кеша тайтлов.

Hosting:

- Vercel Hobby;
- деплой из GitHub;
- переменные окружения через Vercel Dashboard.

External API:

- TMDB API для поиска фильмов, сериалов, постеров, описаний, рейтингов, актеров и жанров.

## 6. Почему выбран Vercel

Vercel удобен для такого проекта, потому что:

- хорошо подходит для Next.js;
- не требует отдельного сервера;
- можно деплоить прямо из GitHub;
- есть бесплатный Hobby-план для личных некоммерческих проектов;
- можно хранить API-ключи в Environment Variables;
- можно быстро дать друзьям ссылку на рабочую версию.

Важное ограничение: Vercel Hobby подходит только для личного некоммерческого использования. Для этого проекта это нормально, потому что оплаты, рекламы и коммерческих целей нет.

## 7. Почему выбран Neon Postgres

Neon подходит для тестового проекта, потому что:

- это настоящий облачный Postgres;
- есть бесплатный план;
- можно подключить к Vercel;
- база может засыпать при неактивности, что экономит ресурсы;
- для 50 пользователей лимитов должно хватить;
- можно использовать pooled connection string для serverless-окружения.

Важное техническое решение:

- `DATABASE_URL` - pooled connection string для приложения;
- `DIRECT_URL` - direct connection string для миграций Prisma.

Это нужно, потому что Vercel использует serverless/serverful-гибридный подход, и большое количество коротких подключений к базе может быть проблемой без connection pooling.

## 8. Почему выбран TMDB

TMDB подходит для некоммерческого MVP, потому что:

- есть API для фильмов, сериалов, актеров и изображений;
- можно получить ключ через аккаунт TMDB;
- API бесплатен для некоммерческого использования при правильной атрибуции;
- данные достаточно хорошие для тестового проекта.

Что нужно добавить в сервис:

- раздел `О проекте` или `Credits`;
- логотип или указание TMDB;
- текст: `This product uses the TMDB API but is not endorsed or certified by TMDB.`

Что важно не делать:

- не выдавать данные TMDB за собственные;
- не использовать сервис коммерчески без проверки условий;
- не копировать всю базу TMDB в свою базу;
- не хранить лишние данные, если их можно получить по API.

## 9. Что входит в MVP

MVP - это первая рабочая версия, которую можно дать друзьям.

### 9.1. Авторизация

Минимально:

- вход через Google;
- создание пользователя в базе при первом входе;
- выход из аккаунта;
- защита личных страниц.

Не делать в первой версии:

- регистрацию по email и паролю;
- восстановление пароля;
- двухфакторную авторизацию;
- роли администратора;
- сложные настройки приватности.

### 9.2. Поиск и каталог

Функции:

- поиск по названию;
- переключатель `Фильмы` / `Сериалы` / `Все`;
- карточки результатов;
- постер;
- название;
- год;
- тип контента;
- рейтинг TMDB;
- краткое описание.

Фильтры первой версии:

- жанр;
- год;
- рейтинг от;
- тип: фильм или сериал.

Автодополнение можно сделать после базового поиска. В первой версии достаточно поиска по кнопке или по Enter.

### 9.3. Детальная страница тайтла

Для фильма:

- постер;
- название;
- оригинальное название;
- год;
- описание;
- жанры;
- рейтинг TMDB;
- актеры;
- режиссер;
- длительность;
- кнопка добавления в список;
- выбор статуса;
- оценка 1-10;
- избранное;
- дата просмотра.

Для сериала:

- постер;
- название;
- оригинальное название;
- год начала;
- описание;
- жанры;
- рейтинг TMDB;
- актеры;
- создатели или режиссеры, если доступны;
- количество сезонов;
- количество эпизодов;
- статус просмотра;
- текущий сезон;
- текущий эпизод;
- кнопка `Следующая серия`.

### 9.4. Трекинг просмотра

Статусы:

- `planned` - запланировано;
- `watching` - смотрю;
- `watched` - посмотрено;
- `dropped` - брошено.

Для фильма:

- статус;
- оценка;
- избранное;
- дата просмотра;
- заметка, опционально.

Для сериала:

- статус;
- текущий сезон;
- текущий эпизод;
- оценка;
- избранное;
- дата последнего обновления прогресса.

Дата просмотра:

- автоматически ставится при переводе в статус `watched`;
- пользователь может изменить дату вручную позже.

### 9.5. Личный список

Страница `Мой список` должна быть одной из главных.

Функции:

- все добавленные фильмы и сериалы;
- фильтр по статусу;
- фильтр по типу;
- фильтр по оценке;
- фильтр по избранному;
- сортировка по дате добавления;
- сортировка по оценке;
- сортировка по названию;
- быстрый переход на детальную страницу;
- быстрые действия: поменять статус, поставить оценку, удалить.

### 9.6. Избранное

Функции:

- кнопка лайка или избранного на карточке;
- фильтр `Избранное` в личном списке;
- избранное хранится отдельно от оценки.

Логика:

- пользователь может поставить фильму 6/10, но добавить в избранное;
- пользователь может оценить фильм 10/10, но не добавлять в избранное.

### 9.7. Пользовательские коллекции

Функции:

- создать коллекцию;
- изменить название;
- добавить описание;
- добавить фильм или сериал;
- удалить тайтл из коллекции;
- открыть страницу коллекции;
- сделать коллекцию приватной или публичной.

Примеры коллекций:

- `Хорроры`;
- `Для вечера`;
- `Смотреть с друзьями`;
- `Фильмы на выходные`;
- `То, что надо пересмотреть`.

В MVP можно сделать коллекции приватными по умолчанию.

### 9.8. Комната выбора фильма

Это главная оригинальная фича первой версии.

Сценарий:

1. Пользователь создает комнату.
2. Выбирает фильтры: жанр, год, рейтинг, тип, длительность.
3. Сервис получает подборку из TMDB.
4. Пользователь копирует ссылку на комнату.
5. Друг открывает ссылку.
6. Каждый участник ставит лайк или дизлайк тайтлам.
7. Сервис показывает совпадения.

Минимальная логика:

- комната имеет владельца;
- гость может войти по ссылке без регистрации, указав имя;
- каждый участник голосует за тайтлы;
- если два или больше участников лайкнули один тайтл, это match;
- matches отображаются отдельным блоком.

Что не делать в первой версии:

- чат внутри комнаты;
- видео-звонки;
- сложные алгоритмы совместимости;
- бесконечную ленту рекомендаций;
- историю всех прошлых комнат на отдельной странице.

## 10. Что не входит в MVP

Эти функции не делаются в первой версии:

- уведомления о новых эпизодах;
- календарь выпусков;
- полноценные друзья и подписки;
- лента активности;
- комментарии и отзывы;
- AI-рекомендации;
- достижения и ачивки;
- подробная статистика;
- публичные рейтинги пользователей;
- мобильное приложение;
- интеграции со стримингами;
- импорт из Letterboxd, IMDb, Trakt или Кинопоиска;
- платные функции;
- реклама.

Причина: первая версия должна проверить ядро продукта, а не пытаться стать большой социальной сетью.

## 13. Модель базы данных

### 13.1. Основной принцип

Не нужно хранить всю информацию о фильмах и сериалах.

Своя база хранит:

- пользователей;
- локальные ссылки на TMDB тайтлы;
- пользовательские статусы;
- оценки;
- избранное;
- прогресс по сериалам;
- коллекции;
- комнаты выбора;
- голоса в комнатах.

Данные TMDB можно подгружать по API. Для скорости можно хранить минимальный кеш: название, постер, год, тип.

### 13.2. Сущности

Основные таблицы:

- `User`;
- `Title`;
- `UserTitle`;
- `SeriesProgress`;
- `Collection`;
- `CollectionItem`;
- `PickRoom`;
- `PickRoomMember`;
- `PickRoomCandidate`;
- `PickVote`.

### 13.3. Prisma schema draft

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")
  directUrl = env("DIRECT_URL")
}

enum TitleType {
  movie
  tv
}

enum WatchStatus {
  planned
  watching
  watched
  dropped
}

enum RoomStatus {
  active
  closed
}

enum VoteValue {
  like
  dislike
  skip
}

model User {
  id              String           @id @default(cuid())
  email           String           @unique
  name            String?
  image           String?
  createdAt       DateTime         @default(now())
  updatedAt       DateTime         @updatedAt

  userTitles      UserTitle[]
  collections     Collection[]
  ownedRooms      PickRoom[]       @relation("RoomOwner")
  roomMembers     PickRoomMember[]
}

model Title {
  id              String           @id @default(cuid())
  tmdbId          Int
  type            TitleType
  title           String
  originalTitle   String?
  posterPath      String?
  releaseYear     Int?
  createdAt       DateTime         @default(now())
  updatedAt       DateTime         @updatedAt

  userTitles      UserTitle[]
  collectionItems CollectionItem[]
  roomCandidates  PickRoomCandidate[]
  votes           PickVote[]

  @@unique([tmdbId, type])
  @@index([title])
}

model UserTitle {
  id              String           @id @default(cuid())
  userId          String
  titleId         String
  status          WatchStatus      @default(planned)
  rating          Int?
  favorite        Boolean          @default(false)
  watchedAt       DateTime?
  note            String?
  createdAt       DateTime         @default(now())
  updatedAt       DateTime         @updatedAt

  user            User             @relation(fields: [userId], references: [id], onDelete: Cascade)
  title           Title            @relation(fields: [titleId], references: [id], onDelete: Cascade)
  seriesProgress  SeriesProgress?

  @@unique([userId, titleId])
  @@index([userId, status])
}

model SeriesProgress {
```

## 14. Основные API-роуты

### 14.1. TMDB proxy

`GET /api/tmdb/search`

Назначение:

- поиск фильмов и сериалов через TMDB;
- скрывает TMDB API key от клиента.

Query params:

- `q`;
- `type`;
- `year`;
- `genre`;
- `ratingFrom`;
- `page`.

`GET /api/tmdb/title`

Query params:

- `type`;
- `tmdbId`.

Назначение:

- получить детали фильма или сериала;
- получить credits;
- подготовить данные для страницы тайтла.

`GET /api/tmdb/genres`

Назначение:

- получить список жанров для фильтров.

### 14.2. Пользовательский список

`GET /api/user-titles`

Назначение:

- получить список тайтлов пользователя;
- применить фильтры и сортировку.

`POST /api/user-titles`

Назначение:

- добавить тайтл в список;
- создать `Title`, если его еще нет в локальной базе;
- создать или обновить `UserTitle`.

`PATCH /api/user-titles/[id]`

Назначение:

- обновить статус;
- обновить оценку;
- обновить избранное;
- обновить дату просмотра;
- обновить заметку.

`DELETE /api/user-titles/[id]`

Назначение:

- удалить тайтл из списка пользователя.

### 14.3. Прогресс сериалов

`PATCH /api/user-titles/[id]/progress`

Назначение:

- обновить сезон;
- обновить эпизод;
- обработать кнопку `Следующая серия`.

### 14.4. Коллекции

`GET /api/collections`

Назначение:

- получить коллекции пользователя.

`POST /api/collections`

Назначение:

- создать коллекцию.

`PATCH /api/collections/[id]`

Назначение:

- изменить название;
- изменить описание;
- изменить публичность.

`DELETE /api/collections/[id]`

Назначение:

- удалить коллекцию.

`POST /api/collections/[id]/items`

Назначение:

- добавить тайтл в коллекцию.

`DELETE /api/collections/[id]/items/[itemId]`

Назначение:

- удалить тайтл из коллекции.

### 14.5. Комнаты выбора

`POST /api/rooms`

Назначение:

- создать комнату;
- сохранить фильтры;
- получить кандидатов из TMDB;
- сохранить кандидатов в базу.

`GET /api/rooms/[id]`

Назначение:

- получить комнату;
- получить участников;
- получить кандидатов;
- получить совпадения.

`POST /api/rooms/[id]/join`

Назначение:

- войти в комнату как зарегистрированный пользователь или гость.

`POST /api/rooms/[id]/vote`

Назначение:

- сохранить лайк, дизлайк или пропуск.

`GET /api/rooms/[id]/matches`

Назначение:

- вернуть тайтлы, которые лайкнули два или больше участников.

## 15. Пользовательские сценарии

### 15.1. Пользователь добавляет фильм в список

1. Пользователь открывает страницу сериала.
2. Добавляет сериал в список.

4. Пользователь отправляет ссылку другу.
5. Друг вводит имя.
6. Оба свайпают или нажимают лайк/дизлайк.
7. Сервис показывает совпадение.
8. Пользователи выбирают фильм для просмотра.

## 20. Логика работы с TMDB

### 20.1. Что получать из TMDB

Для поиска:

- id;
- media_type;
- title или name;
- release_date или first_air_date;
- poster_path;
- vote_average;
- overview;
- genre_ids.

Для детальной страницы:

- название;
- оригинальное название;
- описание;
- жанры;
- постер;
- backdrop;
- дата выхода;
- длительность фильма;
- количество сезонов и эпизодов для сериала;
- рейтинг;
- credits.

### 20.2. Что хранить локально

Минимально:

- tmdbId;
- type;
- title;
- originalTitle;
- posterPath;
- releaseYear.

Не хранить в первой версии:

- все актерские составы;
- все постеры;
- все описания;
- все эпизоды;
- все отзывы;
- большую копию TMDB.

### 20.3. Почему нужен server-side запрос

TMDB API key нельзя отдавать в браузер.

Правильная схема:

```text
browser -> /api/tmdb/search -> TMDB API
```

Неправильная схема:

```text
browser -> TMDB API напрямую с открытым ключом
```

## 21. Логика комнаты выбора

### 21.1. Создание комнаты

Владелец выбирает:

- название комнаты;
- тип: фильм, сериал, все;
- жанры;
- год;
- рейтинг от;
- длительность до;
- количество кандидатов, например 20.

Система:

1. Создает комнату.
2. Получает подборку из TMDB.
3. Создает локальные `Title` для кандидатов.
4. Создает `PickRoomCandidate`.
5. Показывает ссылку для приглашения.

### 21.2. Вход гостя

Гость открывает ссылку.

Если он не авторизован:

- вводит имя;
- создается `PickRoomMember` с `guestName`;
- он может голосовать.

Если он авторизован:

- создается `PickRoomMember` с `userId`;
- имя можно взять из профиля.

### 21.3. Голосование

Для каждого кандидата участник выбирает:

- like;
- dislike;
- skip.

Голос хранится в `PickVote`.

Один участник может иметь только один голос на один тайтл.

### 21.4. Расчет совпадений

Минимальная логика:

```text
match = title, у которого count(like) >= 2
```

Если в комнате 2 человека:

```text
match = оба поставили like
```

Если в комнате больше 2 человек:

```text
match = минимум 2 like
```

Позже можно добавить режимы:

- совпадение всех участников;
- совпадение большинства;
- приоритет по среднему рейтингу;
- исключать фильмы, которые кто-то уже смотрел.

## 22. Метрики теста

Тест не должен оцениваться по принципу `всем понравилось`.

Нужны действия.

Минимальные цели:

- 30 друзей зарегистрировались;
- 20 друзей добавили хотя бы 5 тайтлов;
- 10 друзей вернулись через 3 дня;
- 8 друзей использовали комнату выбора;
- 5 друзей сами сказали, что хотят пользоваться дальше;
- 3 друга попросили добавить конкретную функцию.

Хороший знак:

- люди сами добавляют фильмы без просьбы;
- люди возвращаются на сайт;
- люди используют комнату не один раз;
- люди спрашивают, когда будет новая функция;
- люди делятся ссылкой другим.

Плохой знак:

- зарегистрировались и забыли;
- никто не добавляет фильмы;
- комнату попробовали один раз и больше не используют;
- друзья говорят `прикольно`, но не возвращаются;
- все функции нужны только тебе, а не пользователям.

- не добавлять AI до теста;
- не делать социальную сеть до теста;
- не делать уведомления до теста;
- каждую новую функцию спрашивать: она нужна для первой проверки или нет.

### Риск 3. Проблемы с API

Что делать:

- использовать TMDB только через серверные роуты;
- кешировать минимальные данные;
- обрабатывать ошибки;
- не делать слишком много запросов на каждый экран;
- добавить fallback-состояния в UI.

### Риск 4. Проблемы с базой на serverless

Что делать:

- использовать pooled connection string;
- не создавать PrismaClient на каждый запрос без singleton-паттерна;
- использовать `DIRECT_URL` только для миграций;
- не делать тяжелые запросы без индексов.

### Риск 5. Бесплатные лимиты

Что делать:

- следить за лимитами Vercel;
- следить за Neon usage;
- не хранить лишние данные;
- не использовать большие изображения через свой сервер;
- брать изображения по URL TMDB.

## 26. Минимальный дизайн

Стиль:

- темная тема или светлая тема, выбрать одну на старт;
- крупные постеры;
- простые карточки;
- понятные статусы;
- минимум лишнего текста;
- адаптивная верстка под телефон.

Цветовые статусы:

- `Запланировано` - нейтральный;
- `Смотрю` - активный;
- `Посмотрено` - завершенный;
- `Брошено` - приглушенный.

Важно:

- сайт должен нормально открываться на телефоне;
- друзья, скорее всего, будут тестировать с телефона;
- комнату выбора особенно важно делать mobile-first.

## 29. Definition of Done для MVP

MVP можно считать готовым, если:

- сайт открыт по ссылке Vercel;
- пользователь может войти;
- поиск работает;
- детальная страница работает;
- фильм можно добавить в список;
- сериал можно добавить в список;
- можно менять статус;
- можно поставить оценку;
- можно отметить избранное;
- можно сохранять прогресс сериала;
- можно создать коллекцию;
- можно создать комнату выбора;
- друг может зайти по ссылке в комнату;
- голоса сохраняются;
- совпадения показываются;
- есть страница `О проекте` с атрибуцией TMDB;
- `.env` ключи не попали в GitHub;
- проект пережил тест 5-10 пользователей без критичных ошибок.

## 30. Финальная рекомендация

Не начинать с красивого дизайна и AI.

Начинать с самого полезного ядра:

1. Поиск.
2. Страница тайтла.
3. Добавить в список.
4. Мой список.
5. Прогресс сериала.
6. Комната выбора.

Если эти функции будут удобными, проект уже можно дать друзьям. Все остальное лучше добавлять только после реального теста.
